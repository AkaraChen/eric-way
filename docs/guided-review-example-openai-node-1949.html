Thesis

The PR changes runTools from purely sequential tool-call handling to default concurrent execution, while preserving tool result order and retaining sequential behavior when parallel_tool_calls: false or a single function is forced.

Suggested Reading Order

src/lib/AbstractChatCompletionRunner.ts

Start here. This is the behavior change: runToolCall, the sequential branch, the concurrent Promise.allSettled branch, ordered message insertion, and error surfacing.

src/lib/ChatCompletionRunner.ts and streaming runner

These files show the thin adapter change: both runner types now pass the concrete runner into shared tool execution so callbacks receive the expected runner object.

tests/lib/ChatCompletionRunFunctions.test.ts

Read the new tests after the implementation. They prove concurrent start, stable result ordering, and sequential execution when parallel_tool_calls is false.

helpers.md

Finish with docs. The public helper documentation now states the default concurrency behavior and the sequential opt-out.

Line Map

Visible line

Faster grouped tool calls

When the model returns multiple function tool calls, runTools starts them concurrently by default instead of awaiting each one before starting the next.

State line

Order is still deterministic

Concurrent calls can resolve in any order, but the runner adds tool result messages in the original assistant tool-call order before sending the next request.

Compatibility line

Sequential mode remains available

parallel_tool_calls: false keeps tool execution sequential. A forced single function still returns after the first successful function call.

Hidden line

Side effects must settle before error exit

The concurrent path uses Promise.allSettled before throwing a tool error, reducing the chance that unresolved tool side effects continue after the runner has already failed.

Test line

Concurrency and ordering are directly tested

The new regression test starts two tools, resolves the second first, and asserts that final tool messages still arrive as first-call then second-call.

Docs line

Behavior is documented for users

helpers.md now explains default concurrent execution, ordered result delivery, and the parallel_tool_calls: false escape hatch.

Key Risks

No blocker found

The main behavior is implemented in one shared runner path, both non-streaming and streaming adapters call that path, and tests cover the highest-risk observable behavior: concurrent start plus ordered result submission.

Should verify

The PR body calls out waiting for all concurrent calls to settle before surfacing an error. The implementation does this with Promise.allSettled, but the shown diff does not include a direct regression test where one tool rejects while another is still pending.

Residual risk

Concurrency changes the timing of user-provided tool functions. If callers relied on implicit serial side effects without setting parallel_tool_calls: false, behavior can change even though message ordering is preserved.

Follow-up

A small rejection-order test would make the hidden line explicit: start two tools, reject one, keep the other pending briefly, and assert the runner waits before exposing the failure.

Questions For The Author

  • Should the error-settling behavior be covered by a test, since it is called out as a safety property in the PR description?
  • Is there any public migration note needed for callers whose tools mutate shared state and previously depended on serial execution by default?
  • Was the streaming runner exercised manually or by existing shared tests after changing the callback runner object?

Review Recommendation

Comment-only / likely approve after confirmation. The core change is coherent and the main regression path is tested. I would ask for one extra test or an explicit author confirmation around rejection settling before treating the hidden safety line as fully proven.
